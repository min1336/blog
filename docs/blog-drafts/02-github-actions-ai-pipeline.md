# GitHub Actions로 AI 코드리뷰 파이프라인 만들기: 이슈부터 배포까지 자동화

혼자 개발하면 코드 리뷰를 받을 수 없다. 그래서 AI에게 리뷰를 맡기는 파이프라인을 만들었다. 이슈 생성부터 코드 리뷰, 피드백 자동 수정, 배포까지 13개의 GitHub Actions 워크플로우가 연결된 자동화 시스템이다.

## 문제: 혼자 개발하면 놓치는 것들

개인 프로젝트의 가장 큰 약점은 **리뷰어가 없다**는 것이다. 코드를 작성하고 바로 merge하면:
- 보안 취약점을 놓치기 쉽다
- 비효율적인 코드가 그대로 남는다
- "동작하니까 넘어가자"는 습관이 생긴다

회사에서는 동료가 리뷰해주지만, 혼자 만드는 프로젝트에서는 이 역할을 무언가가 대신해야 한다.

## 설계: 4단계 자동화 파이프라인

전체 파이프라인은 4단계로 나뉜다:

```
1단계: 이슈 → 구체화
   이슈 생성 → 자동 분류(라벨링) → 스펙 구체화 → 팀 토론

2단계: 계획 → 구현
   구현 계획 수립 → AI 자동 구현 → PR 생성

3단계: 리뷰 → 수정
   역할별 병렬 리뷰 → 피드백 자동 수정 → 재리뷰

4단계: 검증 → 배포
   CI 테스트 → Docker 빌드 → EC2 배포 → 헬스체크
```

사람이 개입하는 지점은 딱 2곳: **이슈 생성**과 **PR 승인**. 나머지는 전부 자동이다.

## 워크플로우별 역할

### 이슈 자동 분류 (issue-triage.yml)

이슈가 생성되면 제목과 내용을 분석해서 자동으로 라벨을 붙인다.

```yaml
on:
  issues:
    types: [opened]
```

"로그인이 안 돼요" → `bug`, `auth` 라벨
"다크모드 추가해주세요" → `feature`, `frontend` 라벨

수동으로 라벨을 고르는 시간을 없앴다.

### 이슈 스펙 구체화 (issue-enrich.yml)

라벨이 붙은 이슈를 더 구체적인 스펙으로 다듬는다. 모호한 이슈를 구현 가능한 수준으로 구체화하는 것이 목적이다.

예시:
```
Before: "검색 기능 개선"
After:  "검색 쿼리에 카테고리 필터 추가. PostsService.findAll()에
         category 파라미터 추가, QueryBuilder에 WHERE 조건 추가.
         프론트엔드 SearchInput 컴포넌트에 카테고리 드롭다운 추가."
```

### 팀 토론 (team-discuss.yml)

가장 재미있는 워크플로우다. AI가 **여러 역할**로 나뉘어서 토론한다:

| 역할 | 관점 |
|------|------|
| Architect | 아키텍처, 확장성, 기술 부채 |
| Security | 보안 취약점, 인증/인가 |
| QA | 테스트 케이스, 엣지 케이스 |
| Designer | UI/UX, 접근성 |

한 사람이 놓치는 것을 다른 관점에서 잡아준다. 실제로 이 토론에서 "댓글 삭제 시 비밀번호 검증이 타이밍 공격에 취약할 수 있다"는 보안 관점 피드백이 나온 적이 있다.

### PR 코드 리뷰 (pr-review.yml)

PR이 생성되면 역할별로 **병렬 리뷰**가 시작된다:

```
PR 생성
  ├→ Architecture 리뷰 (동시)
  ├→ Security 리뷰 (동시)
  ├→ QA 리뷰 (동시)
  └→ Code Quality 리뷰 (동시)
```

각 리뷰어가 독립적으로 코드를 분석하고 코멘트를 남긴다. 직렬로 실행하면 시간이 오래 걸리지만, 병렬로 실행하면 가장 느린 리뷰 시간만큼만 걸린다.

### 피드백 자동 수정 + 재리뷰 (pr-auto-fix.yml)

이 부분이 가장 삽질이 많았다. 리뷰 코멘트를 분석해서 코드를 자동 수정하고, 수정 후 다시 리뷰를 받는 루프를 만들었다.

```
리뷰 코멘트 → 분석 → 코드 수정 → 커밋 → 재리뷰 요청
                                          ↓
                                    문제 없으면 종료
                                    문제 있으면 → 다시 수정
```

처음에는 무한 루프에 빠지는 문제가 있었다. 봇이 수정한 커밋이 다��� 리뷰를 트리거하고, 리뷰가 또 수정을 트리거하는 식이었다. 이걸 해결하기 위해:
- 수정 횟수 제한 (최대 3회)
- sender 필터링 (봇의 코멘트는 무시)
- `repository_dispatch`로 트리거를 분리

### CI 실패 자동 수정 (ci-auto-fix.yml)

빌드나 테스트가 실패하면 에러 로그를 분석해서 자동으로 수정을 시도한다. 린트 에러나 타입 에러 같은 단순한 실패는 대부분 자동으로 고쳐진다.

## 안전장치

자동화가 많아지면 위험도 커진다. AI가 잘못된 코드를 자동으로 merge해버리면 프로덕션이 깨질 수 있다. 그래서 5가지 안전장치를 넣었다:

1. **PR 승인 필수** — 아무리 AI가 리뷰해도 최종 merge는 사람이
2. **수정 횟수 제한** — 자동 수정 무한루프 방지
3. **헬스체크** — 배포 후 API가 정상 응답하는지 확인
4. **라벨 기반 제어** — 특정 라벨이 붙은 이슈만 파이프라인 진행
5. **concurrency 제한** — 같은 PR에 대한 워크플로우 중복 실행 방지

## 실제 효과

이 파이프라인을 도입한 후:
- PR당 평균 3-4개의 리뷰 코멘트가 자동 생성된다
- 그 중 약 60%는 실제로 유의미한 개선 포인트다
- 보안 관련 피드백은 혼자서는 절대 놓치지 않게 됐다
- `console.log` 같은 디버그 코드가 merge되는 일이 사라졌다

## 한계와 배운 점

솔직히 AI 리뷰가 시니어 개발자의 리뷰를 대체하지는 못한다. 비즈니스 맥락을 이해하지 못하고, 때로는 불필요한 리팩토링을 제안하기도 한다.

하지만 혼자 개발하는 상황에서 **"최소한의 품질 관문"** 역할은 충분히 한다. "누군가 내 코드를 보고 있다"는 사실만으로도 코드를 더 신중하게 작성하게 된다.

기술적으로 가장 많이 배운 것은 **GitHub Actions의 이벤트 시스템**이다. `push`, `pull_request`, `issue_comment`, `repository_dispatch` 등 다양한 트리거를 조합해서 복잡한 워크플로우를 만들 수 있다는 것을 실전으로 배웠다.

## 워크플로우 전체 목록

| 워크플로우 | 트리거 | 역할 |
|-----------|--------|------|
| issue-triage.yml | 이슈 생성 | 자동 분류 및 라벨링 |
| issue-enrich.yml | 라벨 부착 | 스펙 구체화 |
| issue-enrich-reply.yml | 이슈 코멘트 | 후속 질문 처리 |
| team-discuss.yml | 스펙 완료 | 역할별 팀 토론 |
| issue-plan.yml | 토론 완료 | 구현 계획 수립 |
| claude-implement.yml | 계획 승인 | AI 자동 구현 |
| pr-review.yml | PR 생성 | 병렬 코드 리뷰 |
| pr-auto-fix.yml | 리뷰 코멘트 | 피드백 자동 수정 |
| ci-auto-fix.yml | CI 실패 | 빌드/테스트 자동 수정 |
| deploy.yml | main merge | Docker 빌드 → EC2 배포 |
| weekly-maintenance.yml | 매주 스케줄 | 자동 유지보수 |
